# Deployment Guide: Type 1 Diabetes Screening System on Render

## Overview
This guide will help you deploy your Type 1 Diabetes screening system to Render, making it accessible via a public URL.

---

## Prerequisites
- GitHub account (free)
- Render account (free - sign up at https://render.com)
- Your project code

---

## Step 1: Push Your Code to GitHub

### 1.1 Initialize Git Repository (if not already done)
```bash
cd d:\Earlier Suspiction of type 1 diabetes system
git init
git add .
git commit -m "Initial commit: Type 1 Diabetes screening system"
```

### 1.2 Create a GitHub Repository
1. Go to https://github.com/new
2. Create a new repository named `type1-diabetes-screening`
3. **Do NOT** initialize with README (you already have commits)
4. Click "Create repository"

### 1.3 Push to GitHub
```bash
git remote add origin https://github.com/YOUR_USERNAME/type1-diabetes-screening.git
git branch -M main
git push -u origin main
```

Replace `YOUR_USERNAME` with your actual GitHub username.

---

## Step 2: Set Up Render Deployment

### 2.1 Sign Up for Render (if needed)
- Go to https://render.com
- Sign up with GitHub account (recommended for easy integration)

### 2.2 Create a New Web Service
1. Dashboard → **New +** → **Web Service**
2. Select your GitHub repository (`type1-diabetes-screening`)
3. Grant Render permission to access your GitHub repo

### 2.3 Configure the Service
- **Name**: `type1-diabetes-screening` (or your preferred name)
- **Environment**: `Python 3`
- **Build Command**: `pip install --upgrade pip && pip install -r requirements.txt`
- **Start Command**: `python app.py`
- **Plan**: Free tier (for testing)

### 2.4 Set Environment Variables
In the Render dashboard, go to **Environment**:

Add these variables:
```
FLASK_ENV = production
SECRET_KEY = (Render will auto-generate a secure key)
```

Click **Deploy** and wait for the build to complete (2-5 minutes).

---

## Step 3: Access Your Deployed Application

Once deployment is complete, Render will provide a unique URL like:
```
https://type1-diabetes-screening.onrender.com
```

You can now access your application from anywhere!

## Troubleshooting

### Issue: Build fails / Application crashes
**Solution:**
1. Check Render's **Logs** tab for error messages
2. Verify all packages are in `requirements.txt`
3. Ensure `Procfile` has correct syntax

### Issue: File not found errors (patient_records.xlsx, users.xlsx)
**Solution:**
Files are created automatically on first use. The app creates them if they don't exist.

### Issue: HTTPS warnings
**Solution:**
Render provides free SSL certificates automatically. Warnings should disappear after 24 hours.

### Issue: Slow initial load (cold start)
**Solution:**
Free tier Render instances sleep after 15 minutes of inactivity. First request will take 30-60 seconds.
Upgrade to **Paid** tier for always-on service (recommended for production).

## Important Notes for Production

### Security
- ✅ `SECRET_KEY` is auto-generated by Render
- ✅ HTTPS is enabled by default
- ⚠️ Excel files (`patient_records.xlsx`, `users.xlsx`) are stored on the server's ephemeral storage
  - **They will be lost when the server restarts!**

### Data Persistence (Recommended for Production)
For persistent data storage, consider:
1. **PostgreSQL Database** - Render offers free PostgreSQL
2. **AWS S3** - For file storage
3. **MongoDB Atlas** - NoSQL database

To upgrade to persistent storage:
- Create a Redis/PostgreSQL service on Render
- Modify your app to use database instead of Excel files

### Scaling
- **Free tier**: 0.5 CPU, 512 MB RAM (fine for testing)
- **Upgrade paths**: Starter ($7/month), Standard ($25/month), Pro

## Common Deployment Scenarios

### Scenario 1: Local Testing
```bash
python app.py
# Opens at http://localhost:PORT
```

### Scenario 2: Deploy with Data Backup
Before deploying, backup your Excel files:
```bash
git add patient_records.xlsx users.xlsx
git commit -m "Backup patient records"
git push
```

### Scenario 3: Multiple Environments
Create multiple Render services:
- **Development**: Auto-deploy from `develop` branch
- **Production**: Manual deploy from `main` branch
